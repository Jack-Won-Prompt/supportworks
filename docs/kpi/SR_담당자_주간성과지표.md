# SR 담당자별 주간 성과 지표 명세서

> supportworks(WITHWORKS 자체 파생 시스템)에서 SR(Service Request) 담당자의 주간 성과를 측정·관리하기 위한 KPI 명세서입니다. **데이터 소스는 두 가지(SR 요청 실적 + Git 처리 실적)만** 사용합니다. fulfillment 시스템의 [개발 난이도 표](../../fulfillment/docs/WITHWORKS_기획서.md#13-개발-난이도-vibe-coding-환경-기준-분석용-기초-데이터)를 SR 가중치로 활용합니다.

---

## 1. 문서 개요

### 1.1 목적

- SR 담당자의 **주간 생산성·기여도·완료 품질**을 정량 지표로 측정
- 담당자별 SR 배정의 균형 점검·다음 주 계획 수립
- 누적 데이터로 분기·연간 평가 입력 자료 제공

### 1.2 범위

| 항목 | 포함 | 비고 |
|---|:-:|---|
| SR 처리 실적 (배정·완료·난이도 가중) | ✅ | supportworks SR 데이터 |
| Git 처리 실적 (커밋·LOC·파일) | ✅ | withworks 저장소 (단일) |
| 코드 리뷰·테스트 커버리지 | ❌ | 범위 외 |
| AI 호출 로그·세션 데이터 | ❌ | 범위 외 |
| 근태·출퇴근·회의 참여 | ❌ | 범위 외 |

### 1.3 용어 정의

| 용어 | 정의 |
|---|---|
| **SR** | Service Request — supportworks에 등록된 작업 요청(요구사항·이슈·버그·개선·릴리스 작업 등). 담당자가 배정되어 처리 |
| **담당자(Assignee)** | SR에 배정된 supportworks 사용자. 1 SR ↔ 1 Primary Assignee 원칙 (필요 시 보조 담당자 별도 트랙) |
| **난이도 점수** | fulfillment 시스템의 5점 척도 난이도 (1~5). 본 명세서는 SR 가중치로 이 점수를 사용 |
| **가중 처리량(Weighted Throughput)** | Σ (완료 SR의 난이도 점수). 단순 건수가 아닌 난이도 가중 합산 |
| **주간(Week)** | 월요일 00:00 ~ 일요일 23:59 (KST). ISO Week 사용 (예: 2026-W21) |
| **Git 활동(Git Activity)** | 담당자 commit author 이름·이메일로 식별되는 커밋 행위 |

## 2. 데이터 소스

본 KPI는 다음 **두 가지 소스만** 사용합니다.

### 2.1 SR 요청 실적 (supportworks)

| 항목 | 출처 | 비고 |
|---|---|---|
| SR ID·제목·상태 | supportworks DB | SR 테이블 |
| SR 배정 담당자 | supportworks DB | assignee_id |
| SR 난이도 점수 | 본 명세서 §3.2 매핑 규칙 | 1~5 |
| SR 생성일·배정일·완료일 | supportworks DB | created_at, assigned_at, completed_at |

### 2.2 Git 처리 실적

| 항목 | 출처 | 비고 |
|---|---|---|
| 커밋 SHA·메시지·일자 | Git 저장소 (`git log`) | 운영팀 지정 저장소 |
| commit author 이름·이메일 | `git log --format=%an %ae` | 담당자 식별 |
| 변경 파일·LOC | `git log --shortstat` | 추가/삭제 라인 |
| SR 번호 매핑 (선택) | 커밋 메시지 패턴 | `[SR-1234]` 등 컨벤션 |

**측정 범위**: **모든 커밋을 카운트**합니다 (로컬 브랜치 포함). origin/main 머지 기준이 아닌 작성한 커밋 전부 측정. 시행착오·재시도까지 모두 포함되므로 동일 파일 반복 수정에 대한 누진 페널티가 §5.4.1 #1에 강하게 적용됩니다.

**머지 정책**: Merge commit (커밋 보존) — squash 머지 안 함. 모든 커밋이 origin/main에 그대로 보존되어 추적 가능.

**대상 저장소**: `e:/xampp/htdocs/withworks` — **단일 저장소**. 저장소별 분리·분포 지표는 사용하지 않습니다 (KPI는 SR + Git 두 축만으로 측정).

**제외**: 위 두 소스 외 데이터(코드 리뷰, AI 로그, 회의록 등)는 사용하지 않습니다.

## 3. SR 처리 실적 지표

### 3.1 SR 라이프사이클

```
신규 등록 (created)
   ↓
담당자 배정 (assigned)
   ↓
진행 중 (in_progress)
   ↓
완료 (completed)
   ↓ (재오픈 시 다시 in_progress)
종결 (closed)
```

상태 코드는 supportworks SR 모델 정의를 따릅니다. 본 명세서의 KPI는 **`completed` 또는 `closed` 상태 진입을 "완료"로 계산**합니다.

### 3.2 난이도 점수 매핑

각 SR은 등록 시 또는 분석 시점에 fulfillment 난이도 표(58 단위, 1~5점)의 어느 단위에 해당하는지 매핑됩니다.

**매핑 규칙**:
1. SR 등록자가 작업 대상 단위 1개를 선택 (예: `4. 입고 확정` → 3점)
2. 여러 단위에 걸치는 SR은 **최고 난이도** 단위의 점수 적용 (예: 적치(4) + 마스터(2) → 4)
3. 난이도 표에 매핑되지 않는 신규 영역은 평가 담당자가 임시 점수 부여(1~5), 다음 평가 사이클에 표에 추가
4. 매핑 변경은 SR 완료 전까지 가능, 완료 후 변경 시 운영팀 승인 필요

**난이도 표 참조**: [fulfillment/docs/WITHWORKS_기획서.md §13.3 통합 난이도 리스트](../../fulfillment/docs/WITHWORKS_기획서.md#133-통합-난이도-리스트-vibe-coding-환경-재평가)

| 점수 | 등급 | 의미 |
|---:|---|---|
| 1 | ⚪ 매우 낮음 | 표준 CRUD·정적 자원 |
| 2 | ⚪ 낮음 | 자체 패턴 + 표준 패턴 |
| 3 | 🟢 중간 | 다중 모델·트랜잭션 + 자체 시스템 |
| 4 | 🟡 높음 | 도메인 룰·알고리즘·무결성 |
| 5 | 🔴 매우 높음 | 외부 API + 규제 |

### 3.3 SR 지표 (담당자별 주간 단위)

| 지표 | 정의 | 산식 |
|---|---|---|
| **SR 배정 건수** | 주간 동안 담당자에게 배정된 SR 수 (배정일 기준) | `COUNT(SR WHERE assignee=A AND assigned_at IN week)` |
| **SR 완료 건수** | 주간 동안 완료된 SR 수 (완료일 기준) | `COUNT(SR WHERE assignee=A AND completed_at IN week)` |
| **가중 처리량** | 완료 SR의 난이도 점수 합 | `Σ score_i WHERE completed_at IN week` |
| **평균 난이도** | 완료 SR의 평균 난이도 | `가중 처리량 / 완료 건수` |
| **완료율** | 배정 대비 완료 비율 | `완료 건수 / 배정 건수` |
| **평균 처리 시간** | 배정→완료 소요 시간 평균 | `AVG(completed_at - assigned_at)` |
| **이월 SR** | 주간 종료 시점에 진행 중인 SR | `COUNT(SR WHERE assignee=A AND status IN ('assigned','in_progress'))` |
| **재오픈 SR** | 한 번 완료 후 재오픈된 SR | `COUNT(SR WHERE reopen_count >= 1 AND assignee=A)` |

## 4. Git 처리 실적 지표

### 4.1 담당자 ↔ Git 사용자 매핑

각 supportworks 담당자는 1개 이상의 **Git 사용자 식별자**(commit author name 또는 email)와 매핑됩니다.

| 필드 | 예시 |
|---|---|
| supportworks user_id | 42 |
| git_author_emails | `jackwon2021@gmail.com`, `wonsk@company.com` |
| git_author_names | `JackWon`, `원성기`, `Won Sungki` |

매핑은 운영팀 관리. 변경 시 누적 통계 재계산 정책 필요.

### 4.2 Git 지표 (담당자별 주간 단위)

| 지표 | 정의 | git 명령 |
|---|---|---|
| **커밋 수** | 주간 동안 author로 등록된 커밋 수 (머지 커밋 제외) | `git log --no-merges --author=<A> --since=<MON> --until=<SUN> --oneline | wc -l` |
| **추가 LOC** | 추가된 코드 라인 합 | `git log --no-merges --author=<A> --since=<MON> --until=<SUN> --shortstat` 합산 |
| **삭제 LOC** | 삭제된 코드 라인 합 | (동일) |
| **순 변경 LOC** | 추가 - 삭제 | 산식 |
| **변경 파일 수** | 누적 변경 파일 수 | `--shortstat` 합산 |
| **SR 연결 커밋 수** | 커밋 메시지에 `[SR-<id>]` 등 SR 번호 포함된 커밋 | `grep -E '\[SR-[0-9]+\]'` 필터 |

### 4.3 SR ↔ Git 연결 (권장 컨벤션)

커밋 메시지 prefix로 SR 번호 명시 시 추적 가능:

```
[SR-1234] 입고 확정 시 LOT 생성 누락 수정

ModInv::createLot() 호출 누락. 부분 입고 케이스에서 reproduction.
```

매핑 규칙:
- 1 커밋 = 1 SR (1:1) — `[SR-1234]` 한 개만
- 1 커밋 = 다중 SR — `[SR-1234][SR-1235]` 콤마 또는 brackets 연속
- prefix 없는 커밋 = "비-SR 작업" (리팩터링·환경설정 등)으로 분류

### 4.4 제외 규칙

다음 커밋은 통계에서 제외:
- 머지 커밋 (`--no-merges`)
- 리버트 커밋 — 단, 리버트 대상 원 커밋도 차감 옵션
- 자동 생성 커밋 (CI·번역 동기화 등) — author email 화이트리스트로 필터
- 본인 커밋이 아닌 cherry-pick (author 보존 시 정상 카운트)

### 4.5 Commit Efficiency (SR당 커밋 효율)

**배경**: SR 담당자가 Claude AI를 활용해 업무를 처리하는 환경에서는 코드 생성 속도가 빨라지지만, AI 출력의 반복 시도·시행착오가 커밋 수를 부풀릴 수 있습니다. 단순 커밋 수가 노력·생산성 신호로 부정확해지므로 **SR당 커밋 수의 효율성**을 별도 지표로 추적합니다.

| 지표 | 정의 | 산식 |
|---|---|---|
| **SR당 커밋 수 중앙값** | 완료 SR이 소비한 SR 연결 커밋 수의 중앙값 | `MEDIAN(commits_per_sr WHERE sr in completed_set)` |
| **SR당 평균 커밋 수** | (참고용) | `AVG(commits_per_sr)` |
| **시행착오 비율** | 표준 대비 3배 초과 SR 비율 | `COUNT(commits_per_sr > 3 × team_median) / COUNT(completed_sr)` |

**해석 기준** (운영 데이터로 보정):

| SR당 커밋 수 | 의미 |
|---:|---|
| 1~3 | 이상적 — 깔끔한 작업 |
| 4~6 | 정상 범위 — 일반적 디버깅 포함 |
| 7~10 | 시행착오 의심 — 검토 권장 |
| 11+ | 시행착오 강함 — 반복 커밋 페널티 대상 (§5.4 참조) |

본 지표는 점수 직접 반영이 아닌 **보조 신호**로 사용하며, 산출 결과는 §5.4 반복 커밋 페널티의 입력값이 됩니다.

## 5. 통합 KPI 산식

### 5.1 주간 종합 점수 (Weekly Score)

담당자별 주간 종합 점수는 SR과 Git 지표를 가중 결합합니다.

```
WeeklyScore = α × NormalizedSR + β × NormalizedGit

여기서
  NormalizedSR  = 가중 처리량 / 팀 평균 가중 처리량
  NormalizedGit = (커밋 수 × 가중 LOC factor) / 팀 평균
  α + β = 1.0
```

**권장 가중치 (AI 활용 환경 기준)**:
- α (SR 가중) = **0.8**
- β (Git 가중) = **0.2**

근거: SR 담당자가 Claude AI를 활용해 업무를 처리하는 환경에서는 코드 생성 속도가 빨라져 커밋·LOC 절대값이 노력 비례 신호로 부정확해집니다. **결과 품질(SR 완료·난이도)이 진짜 비즈니스 가치 신호**이므로 SR 가중치를 상향(0.8). Git 활동은 보조 신호(과정·기여 흔적)로 사용(0.2).

운영팀이 조직 특성에 따라 가중치 조정 가능 (단, AI 환경에서 β > 0.3은 권장하지 않음).

### 5.2 LOC 가중치 (Diminishing Returns)

순 LOC 증가는 가치와 비례하지 않으므로 로그 스케일 적용:

```
LOC_factor = log10(net_loc + 1)
```

- 100 LOC → 2.00
- 1,000 LOC → 3.00
- 10,000 LOC → 4.01

극단적 LOC가 점수를 왜곡하지 않도록.

### 5.3 품질 보정

다음 항목은 종합 점수에 감점·가점 적용:

| 항목 | 보정 | 산식 |
|---|---|---|
| 재오픈 SR | 감점 | `WeeklyScore -= 재오픈 건수 × 0.5` |
| 평균 처리 시간 (난이도별 표준 대비) | 빠르면 가점, 느리면 감점 | `±(표준시간 - 실제시간) / 표준시간 × 0.1` |
| SR 미매핑 커밋 비율 | 50% 초과 시 경고 (점수 영향 없음) | 알림 |

**난이도별 표준 처리 시간** (참고용 초안 — 운영 데이터로 보정):

| 난이도 | 표준 처리 시간 |
|---:|---|
| 1 | 0.5 일 |
| 2 | 1 일 |
| 3 | 3 일 |
| 4 | 7 일 |
| 5 | 14 일 |

### 5.4 반복 커밋 페널티 (Repeat-Commit Penalty)

**배경**: 본 KPI는 모든 커밋을 카운트하고 머지 정책으로 **Merge commit (커밋 보존)** 을 채택하므로, 시행착오 커밋이 squash로 정리되지 않고 그대로 측정에 반영됩니다. 동일 영역에 반복 수정이 누적되는 것은 작업 단위 설계 부실·재시도·미흡한 사전 점검 신호일 수 있어, **동일 소스 반복 수정에 가장 강한 페널티를 누진적으로 적용**합니다.

#### 5.4.1 임계값과 페널티

| 우선 | 패턴 | 임계·산식 | 페널티 |
|:-:|---|---|---|
| **#1 (최우선·누진)** | **동일 파일 반복 수정** — 같은 파일 경로에 대한 1주 commit 횟수 N | N ≤ 2: 정상 (없음)<br>N = 3~4: 경고만 (점수 영향 없음)<br>N = 5~9: **`-0.10 × (N-4)`** 점<br>N = 10~19: **`-0.50 + (-0.20 × (N-10))`** 점<br>N ≥ 20: **`-2.50` + `(-0.30 × (N-20))`** 점 (강한 누진) | 누진 적용 — 동일 파일 수정이 많을수록 페널티가 지수적으로 증가 |
| #2 | **동일 SR 커밋 폭증** — 1 SR에 연결된 커밋 수가 팀 중앙값의 3배 초과 | 초과 SR 1건당 | `-0.2` 점 |
| #3 | **시행착오 메시지 감지** — 커밋 메시지에 키워드 매칭 (`fix again`, `retry`, `다시`, `재시도`, `수정 v2`, `rollback`, `revert`, `~를 되돌리다`) | 매칭 커밋 1건당 | `-0.1` 점 |

**누진 페널티 예시 (#1 동일 파일 반복 수정)**:

| 1주 commit 횟수 | 페널티 | 등급 |
|---:|---:|---|
| 1~2 | 0 | 정상 |
| 3 | 0 (경고만) | 🔵 경고 |
| 4 | 0 (경고만) | 🔵 경고 |
| 5 | `-0.10` | 🟡 페널티 |
| 7 | `-0.30` | 🟡 페널티 |
| 10 | `-0.70` | 🟠 강한 페널티 |
| 15 | `-1.70` | 🟠 강한 페널티 |
| 20 | `-2.50` | 🔴 매우 강한 페널티 |
| 25 | `-4.00` | 🔴 매우 강한 페널티 |

**산정 단위**: 파일 경로 단위 (예: `app/Http/Controllers/main/medical/standard/ReceivingConfirmController.php`). 같은 파일이라면 SR이 달라도 누적.

#### 5.4.2 페널티 상한

총 페널티는 `WeeklyScore`의 **20%**를 초과하지 않도록 캡(cap) 적용:

```
final_penalty = MIN(raw_penalty, WeeklyScore × 0.2)
```

운영 결정: 50% 차감은 과도하다고 판단하여 **20%로 하향 조정**. 평가 의미는 유지하되 한 차례의 패턴으로 점수가 크게 흔들리지 않도록 위축 효과 추가 방지. 절대 0 이하로는 떨어지지 않음 (`final_score = MAX(0, base_score - final_penalty)`).

#### 5.4.3 보고서 표시

페널티가 발생한 경우 담당자 보고서에 다음 항목 표시:
- 페널티 합계
- 임계 초과 파일·SR·키워드 매칭 커밋 목록 (개선 신호로 활용)

#### 5.4.4 적용 원칙

- **AI 활용 자체는 페널티 대상이 아닙니다 — 비난·평가 절대 금지**. Claude를 활용해 빠르고 정확하게 SR을 완료하는 것은 권장되며, AI 사용 사실 자체가 점수에 부정적 영향을 주지 않습니다.
- 페널티는 **결과 품질의 불안정성**(반복 수정·재시도)에만 적용합니다.
- **단, AI를 활용함에도 능률이 안 나오는 패턴은 별도 신호**로 식별합니다 (§5.5 참조). 이는 평가가 아닌 **개선·코칭 신호**이며, 페널티와는 분리됩니다.
- 일정 수준의 시행착오는 정상이므로 임계값은 보수적으로 설정하고, 운영 데이터로 분기마다 보정합니다.

### 5.5 AI 활용 능률 신호 (Coaching Signal)

**목적**: AI 활용 자체는 비난·평가 대상이 아닙니다. 그러나 **AI를 활용함에도 능률이 안 나오는 패턴**은 담당자가 어려움을 겪고 있다는 신호일 수 있으며, **1:1 코칭·페어 작업·도메인 지식 보강** 등 지원으로 연결되어야 합니다. 본 섹션은 그 신호를 식별하는 규칙입니다.

**원칙**:
- 본 신호는 **평가 지표가 아니라 지원 신호**입니다.
- 신호 발생 = 코칭 권장. 신호 미발생 = 평가에 영향 없음.
- 신호 발생 시 본인에게만 우선 표시(자가 진단). 매니저에게 노출은 본인 동의 또는 임계 누적 후.

#### 5.5.1 능률 신호 입력 지표 (4가지 신호의 조합)

| # | 지표 | 임계 (능률 저조 의심) |
|---:|---|---|
| 1 | Commit Efficiency (SR당 커밋 중앙값) | 팀 중앙값의 **2배 초과** 또는 SR당 7+ |
| 2 | 재오픈 SR 건수 | 주간 **2건 이상** |
| 3 | 평균 처리 시간 | 난이도별 표준의 **2배 초과** |
| 4 | 반복 커밋 페널티 누적 | 주간 페널티 합계 **-0.3 이하** |

#### 5.5.2 능률 신호 등급

4가지 신호 중 충족 개수로 등급 결정:

| 충족 개수 | 등급 | 의미 | 처리 |
|---:|---|---|---|
| 0 | ✅ 양호 | AI 활용 능률 정상 또는 우수 | 표시 없음 |
| 1 | 🔵 관찰 | 일부 신호. 학습 과정 가능성 | 본인에게만 자가 진단 패널 표시 |
| 2 | 🟡 주의 | 능률 저조 가능성 | 본인 자가 진단 + 개선 가이드 제안 |
| 3~4 | 🟠 코칭 권장 | 도메인 지식·작업 방식 지원 필요 | 본인 + 매니저 알림 (본인 동의 시) → 1:1 코칭 워크플로우 |

#### 5.5.3 코칭 워크플로우

능률 신호가 **2 주 연속 🟡 주의** 이상이거나 **단일 주에 🟠 코칭 권장**으로 분류되면 다음 절차 진행:

```
1. 본인에게 자가 진단 패널 표시
   - 어떤 신호가 발생했는지 (Commit Efficiency / 재오픈 / 처리 시간 / 페널티)
   - 일반적인 개선 힌트 (예: "Claude에게 명세서 컨텍스트를 먼저 제공해보세요")
   ↓
2. 본인이 도움 요청 옵션 선택 가능
   [ ] 매니저에게 알림
   [ ] 도메인 멘토 매칭 요청
   [ ] 페어 작업 요청
   [ ] 표시만 (도움 요청 안 함)
   ↓
3. 매니저 알림은 본인 동의 시에만 (개인정보·심리적 안전)
   ↓
4. 코칭 세션 후 능률 신호 추적
   - 다음 주 신호 감소 → 종결
   - 신호 유지 → 다른 형태 지원 (도메인 학습 자료·작업 분할)
```

#### 5.5.4 개선 가이드 (자가 진단 시 자동 제시)

| 발생한 신호 | 자동 제시되는 힌트 |
|---|---|
| Commit Efficiency 초과 | "Claude에게 작업 시작 전 관련 컨트롤러·모델·헬퍼 컨텍스트를 한 번에 제공하면 시행착오를 줄일 수 있습니다. 코드베이스의 자체 패턴(`IntegrationController`, `DataHeader`, `ModInv` 등)을 먼저 설명해 보세요." |
| 재오픈 잦음 | "완료 전에 동일 도메인의 이전 SR·관련 화면을 확인해 부작용을 사전 점검하세요. fulfillment 난이도 표 §13의 자체 시스템 학습 비용 영역(점수 3)은 회귀 영향이 큽니다." |
| 처리 시간 초과 | "난이도 4~5 SR은 외부 사양·도메인 룰 확인이 필수입니다. 시작 전 [WITHWORKS 공통흐름](../../fulfillment/docs/WITHWORKS_기획서_공통흐름.md)·도메인 기획서를 먼저 읽으세요." |
| 반복 커밋 페널티 누적 | "동일 영역 재수정이 잦다면 작업 분할이 너무 크지 않은지 점검하세요. 한 SR을 2~3 커밋 내로 마치는 것을 목표로 단위를 잘게 쪼개 보세요." |

#### 5.5.5 절대 금지 사항

- **AI 사용 자체에 대한 비난·평가·점수 차감** — 본 신호는 AI 사용 여부와 무관합니다.
- **개인 명단 공개 비교** — 능률 신호 등급은 본인 또는 매니저에게만 노출. 팀 전체에 개인 명단으로 비교 게시 금지.
- **자동 페널티** — 능률 신호는 코칭 신호이지 점수 페널티가 아닙니다. WeeklyScore에 직접 차감하지 않습니다.
- **단일 주 데이터 단독 판단** — 최소 2주 추세를 보고 코칭 워크플로우 진입.

## 6. 주간 집계·산출 방식

### 6.1 산출 시점

매주 **월요일 09:00 KST**에 직전 주(월~일) 데이터를 집계.

### 6.2 산출 절차

```
1. 주간 경계 결정 (ISO Week)
   - 시작: 직전 월요일 00:00:00 KST
   - 종료: 직전 일요일 23:59:59 KST

2. SR 데이터 수집
   - supportworks DB에서 assigned_at·completed_at·reopened_at가 주간에 속한 SR 추출
   - 담당자별 그룹핑
   - 난이도 점수 결합

3. Git 데이터 수집
   - withworks 저장소에서 git log 추출
   - 담당자 매핑 (author email/name → user_id)
   - 머지 커밋·자동 커밋 제외
   - 커밋 메시지에서 SR 번호 파싱

4. 지표 계산
   - SR 8개 지표 (§3.3)
   - Git 7개 지표 (§4.2)
   - 통합 점수 (§5.1)

5. 보고서 생성
   - 담당자별 (§7.1)
   - 팀 종합 (§7.2)

6. 배포
   - supportworks 대시보드에 게시 (선택)
   - 주간 요약 이메일 발송 (선택)
```

### 6.3 데이터 정합성 검증

집계 전 다음을 확인:
- supportworks SR 상태 코드 정합성 (open/assigned/in_progress/completed/closed)
- Git author 매핑 누락 여부
- 미매핑 SR (난이도 점수 없음) — 임시 점수 또는 제외
- 시간대(KST) 일관성

## 10. 한계·주의사항

- **단일 지표 의존 위험**: SR 가중 처리량과 Git 커밋만으로 모든 성과를 측정하지 않음. 코드 품질·협업·리뷰 기여는 별도 평가가 필요
- **LOC의 한계**: 라인 수는 가치와 비례하지 않음 (로그 스케일 + 가중치로 완화). 리팩터링·삭제 작업이 저평가될 수 있음
- **난이도 매핑의 주관성**: 1~5 점수 부여에 평가자 편향 가능. 매핑 가이드라인 별도 정립 필요
- **Git author 변경**: 담당자가 이메일·이름 변경 시 누적 통계 분리 위험. 매핑 갱신 절차 필수
- **SR↔Git 1:1 가정**: 1 커밋이 여러 SR을 한 번에 처리하는 경우 가중치 분배 규칙 필요 (현재는 단순 카운트)
- **AI 활용 환경 특수성**: SR 담당자가 Claude AI를 활용해 업무를 처리하므로 단순 커밋·LOC가 노력 비례 신호가 아님. 본 명세서는 SR 비중을 0.8로 상향(§5.1)하고 반복 커밋 페널티(§5.4)를 도입해 보정. 그러나 시행착오와 학습은 정상이므로 페널티는 캡(cap) 적용으로 위축 효과를 제한
- **AI 활용 자체 비측정**: AI 사용은 supportworks 환경의 기본 전제로 별도 측정 안 함. 코드 생산성보다 **결과 품질(SR 완료·재오픈 빈도·SR당 커밋 효율)** 에 집중
- **본 명세서는 측정 도구이지 평가 도구가 아님**: 절대 순위가 아닌 추세 신호로 활용 권장
