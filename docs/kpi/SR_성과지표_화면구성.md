## 1. 화면 개요

### 1.1 대상 화면

| 화면 | 라우트 | 대상 사용자 | 용도 |
|---|---|---|---|
| **본인 주간 KPI** | `/my-weekly` (탭 신규 추가 또는 기존 확장) | 모든 SR 담당자 | §7.1 담당자별 주간 보고서 |
| **팀 종합 KPI** | `/my-weekly` (관리자/매니저 탭 또는 별도 라우트) | 관리자·매니저 | §7.2 팀 종합 보고서 |
| **SR 난이도 매핑** | `/sr/<id>/edit` 또는 SR 상세 인라인 | SR 등록자·담당자 | 1~58 난이도 단위 선택 |

### 1.2 전제

- supportworks 기존 `/my-weekly` 화면을 **확장**하는 방식 (대규모 신규 라우트 도입 지양)
- 기존 탭(보고서 목록, 웍스 서머리)에 **세 번째 탭 "성과 지표"** 추가
- supportworks 디자인 토큰 사용 — `--color-text-primary`, `--t50`/`--t100`/`--t600`/`--t700`, `--color-bg-muted`, `--color-border-default`

### 1.3 권한 (기존 컨트롤러 패턴 재사용)

| 화면 | 접근 권한 |
|---|---|
| 본인 주간 KPI | 모든 로그인 사용자 (본인 데이터만) |
| 팀 종합 KPI | `$user->isAdmin()` 또는 `manager` 역할 보유자 |
| SR 난이도 매핑 편집 | SR 등록자·담당자·관리자 |

### 2.2 컴포넌트별 명세

#### 2.2.1 헤더 (Header)

| 요소 | 동작 |
|---|---|
| 주차 선택 화살표 | 이전·다음 ISO Week 이동. 현재 주차 이후는 비활성 |
| 주차 라벨 (`2026-W21`) | 클릭 시 캘린더 팝업으로 임의 주차 선택 |
| 대상 라벨 | 본인 KPI는 "본인" 고정. 팀 KPI는 담당자 선택 드롭다운 |
| Word 다운로드 | 현재 주차 KPI를 Word로 (`/projects/{project}/weekly-reports/.../download` 패턴 재사용) |

#### 2.2.2 종합 점수 카드 (Hero Card)

| 요소 | 데이터 바인딩 |
|---|---|
| 점수 (큰 글자) | `weekly_kpi_snapshots.weekly_score` (소수점 둘째 자리) |
| 점수 게이지 | 0~3 범위 시각화 (1.00=팀 평균선, 2.00 이상 강조색) |
| 산식 줄 | `SR 0.8 × {normalized_sr} + Git 0.2 × {normalized_git} = {score}` 한 줄 |
| 팀 평균 대비 | `(weekly_score / team_avg - 1) × 100%` |
| 전주 대비 | `current_score - previous_score` (소수점 둘째 자리) |

색상 등급 (점수 게이지 색):
- 0.0 ~ 0.7: `#94a3b8` 회색 (저조)
- 0.7 ~ 1.3: `#3b82f6` 청색 (평균)
- 1.3 ~ 2.0: `#10b981` 녹색 (양호)
- 2.0+ : `#f59e0b` 황색 (탁월)

#### 2.2.3 SR 처리 실적 카드 (8 지표)

데이터 바인딩 (모두 `weekly_kpi_snapshots` 컬럼에서 직접 읽기):

| 라벨 | 컬럼 | 표시 형식 |
|---|---|---|
| 배정 건수 | `sr_assigned` | 정수 |
| 완료 건수 | `sr_completed` | 정수 |
| 가중 처리량 | `weighted_throughput` | 정수 |
| 평균 난이도 | `avg_difficulty` | 소수점 1자리 |
| 완료율 | `completion_rate` | `%`, 소수점 1자리 |
| 평균 처리 시간 | `avg_handling_days` | `일`, 소수점 1자리 |
| 이월 SR | (실시간 계산) | 정수 |
| 재오픈 SR | `sr_reopened` | 정수 |

레이아웃: 데스크탑 8열 그리드, 모바일 4열×2행.

#### 2.2.4 완료 SR 목록 (Table)

| 컬럼 | 데이터 | 정렬 |
|---|---|---|
| SR ID | `sr.id` (링크) | 기본 정렬 |
| 제목 | `sr.title` (말줄임) | — |
| 난이도 | `sr_difficulty_mappings.score` + 등급 뱃지 (1~5 색상) | 클릭 정렬 |
| 처리 시간 | `completed_at - assigned_at` (일 단위) | 클릭 정렬 |
| 상태 | `completed` / `closed` 뱃지 | — |

행 클릭 → SR 상세 팝업 (기존 `openWeeklyPopup` 패턴 활용).

빈 상태: "완료한 SR이 없습니다" 안내.

#### 2.2.5 Git 처리 실적 카드 (7 지표)

| 라벨 | 컬럼 | 표시 형식 |
|---|---|---|
| 커밋 수 | `git_commits` | 정수 |
| 추가 LOC | `git_added_loc` | `+숫자`, 천 단위 콤마 |
| 삭제 LOC | `git_deleted_loc` | `-숫자` |
| 순 LOC | `git_added_loc - git_deleted_loc` | `±숫자` |
| 변경 파일 | `git_files` | 정수 |
| SR 연결 커밋 | `sr_linked_commits` + `(SR연결 / 전체 × 100%)` | `N (X%)` |
| Commit Efficiency | (SR 연결 커밋 / SR 완료 건수)의 중앙값 | `소수점 1자리 commit/SR` |

색상 가이드 (Commit Efficiency):
- 1.0 ~ 3.0: 녹색 (이상적)
- 3.0 ~ 6.0: 회색 (정상)
- 6.0 ~ 10.0: 황색 (시행착오 의심)
- 10.0+ : 적색 (페널티 대상)

#### 2.2.6 품질 보정 패널

| 항목 | 표시 |
|---|---|
| 재오픈 감점 | `재오픈 건수 × 0.5` 음수 |
| 처리 속도 보정 | `±값` + (빠름/느림) 라벨 |
| 미매핑 커밋 비율 | `%` + (양호/주의/경고) — 50% 초과 시 적색 |

알림 패널: 미매핑 SR 또는 처리 속도가 표준 대비 2배 이상 느릴 때 알림 박스 표시.

#### 2.2.7 반복 커밋 페널티 패널 (§5.4 시각화)

**중점**: 본 시스템은 모든 커밋을 카운트하므로 **동일 파일 반복 수정에 누진 페널티**가 가장 강하게 적용됩니다. 시행착오·재시도가 그대로 측정에 반영되므로 작업 단위·사전 점검의 신중함이 중요합니다.

| 요소 | 데이터 |
|---|---|
| 페널티 합계 | `repeat_commit_penalty` (음수, 캡 적용 후 값. 캡 = WeeklyScore × 20%) |
| 임계 초과 항목 리스트 | 다음 3가지 패턴별, **#1을 가장 강조** |
| └ **#1 동일 파일 반복 수정 (누진)** | 파일명 + 수정 횟수 + 적용 페널티 (`-0.10` ~ `-4.00`) + 등급(🔵🟡🟠🔴) |
| └ #2 동일 SR 커밋 폭증 | SR ID + 커밋 수 + 페널티 |
| └ #3 시행착오 키워드 매칭 | 키워드 + 커밋 SHA + 페널티 |

**#1 동일 파일 반복 수정 표시 양식** (가장 큰 시각 비중 부여):

```
┌─────────────────────────────────────────────────────────────────┐
│ 🟠 동일 파일 반복 수정 (강한 페널티 -0.70)                       │
│ app/Http/Controllers/main/medical/standard/                      │
│   ReceivingConfirmController.php                                 │
│ 이번 주 10회 수정 — 작업 단위가 너무 크거나 사전 점검이 미흡할 수 │
│ 있습니다. 단일 SR 분할·사전 영향 분석을 권장합니다.              │
└─────────────────────────────────────────────────────────────────┘
```

등급별 색상:
- 🔵 경고 (3~4회): `--color-text-secondary` (회색) — 정보 신호만
- 🟡 페널티 (5~9회): `#f59e0b` (황색)
- 🟠 강한 페널티 (10~19회): `#f97316` (주황)
- 🔴 매우 강한 페널티 (20회+): `#ef4444` (적색)

페널티 없으면 패널 자체 비표시 (또는 "양호" 녹색 메시지).

#### 2.2.8 AI 활용 능률 신호 패널 (§5.5 시각화) — 본인에게만 표시

**원칙**: 본 패널은 **평가가 아닌 자가 진단·개선 신호**입니다. AI 사용 자체에 대한 비난·평가가 아니며, 점수에 직접 반영되지 않습니다.

```
┌─────────────────────────────────────────────────────────────────┐
│ 🔵 AI 활용 능률 자가 진단 (이번 주)                              │
│                                                                 │
│ 등급: 🟡 주의 (4가지 신호 중 2가지 발생)                         │
│                                                                 │
│ 감지된 신호:                                                    │
│  ⚠ Commit Efficiency 8.2 commit/SR (팀 중앙값 3.6의 2.3배)       │
│  ⚠ 평균 처리 시간 5.2일 (난이도 3 표준 3일의 1.7배)              │
│                                                                 │
│ 💡 개선 힌트:                                                    │
│  • Claude에게 작업 시작 전 관련 컨트롤러·모델·헬퍼 컨텍스트를     │
│    한 번에 제공하면 시행착오를 줄일 수 있습니다.                 │
│  • 자체 패턴(IntegrationController, DataHeader, ModInv) 컨텍스트│
│    을 먼저 설명하세요.                                          │
│                                                                 │
│ 도움 요청 (선택):                                              │
│  [ ] 매니저에게 알림    [ ] 도메인 멘토 매칭    [ ] 페어 작업    │
│  [표시만 — 도움 요청 안 함]                                     │
└─────────────────────────────────────────────────────────────────┘
```

| 요소 | 동작 |
|---|---|
| 등급 뱃지 | ✅ 양호 / 🔵 관찰 / 🟡 주의 / 🟠 코칭 권장 (§5.5.2 산식) |
| 감지된 신호 목록 | 발생한 1~4개 신호 + 실제 값 + 임계 |
| 개선 힌트 | §5.5.4 표에서 신호별 자동 매칭된 힌트 |
| 도움 요청 옵션 | 본인이 선택. **매니저 알림은 본인 동의 필수** |
| 단일 주 데이터 단독 표시 | 등급은 표시하되 코칭 워크플로우 진입은 **2주 연속 🟡 이상** 또는 **단일 주 🟠** |

**표시 규칙**:
- 등급 ✅ 양호 → 패널 자체 비표시
- 🔵 관찰 → 작은 정보 박스만 (개선 힌트 없이)
- 🟡 주의 → 패널 표시 + 개선 힌트
- 🟠 코칭 권장 → 패널 표시 + 개선 힌트 + 매니저 알림 옵션 강조

**절대 금지** (§5.5.5):
- AI 사용 자체에 대한 비난·평가 문구 사용 금지
- 다른 담당자와의 직접 비교 표시 금지 (팀 중앙값 비교는 익명 통계 형태로만)
- 단일 주 능률 신호로 자동 페널티 부과 금지 (WeeklyScore에 직접 차감 X)

### 3.2 컴포넌트별 명세

#### 3.2.1 필터 헤더

| 요소 | 동작 |
|---|---|
| 주차 선택 | 본인 화면과 동일 |
| 회사 드롭다운 | `CompanyGroup` 필터 (기존 `srCompaniesForFilter` 재사용) |
| 프로젝트 드롭다운 | 매니저는 본인 매니저 프로젝트, 관리자는 전체 |

#### 3.2.2 팀 요약 카드 (4개)

데이터: 필터 적용 후 `weekly_kpi_snapshots` 집계

| 카드 | 산식 |
|---|---|
| SR 완료 | `SUM(sr_completed)` + `(가중처리량)` 부가 표시 |
| 총 커밋 | `SUM(git_commits)` |
| 순 LOC | `SUM(git_added_loc - git_deleted_loc)` |
| 활동 인원 | `COUNT(DISTINCT user_id WHERE weekly_score > 0)` |

#### 3.2.3 담당자별 랭킹 표

정렬: `weekly_score DESC` 기본. 컬럼 클릭으로 변경 가능.

| 컬럼 | 데이터 | 표시 |
|---|---|---|
| 순위 | RANK() OVER ORDER BY weekly_score DESC | 1~N |
| 담당자 | `user.name` + avatar | 텍스트 + 이미지 |
| 종합 | `weekly_score` + 전주 대비 ⬆⬇ | 점수 + 변동 화살표 |
| 가중처리 | `weighted_throughput` | 정수 |
| Git 커밋 | `git_commits` | 정수 |
| 페널티 | `repeat_commit_penalty` | 음수 (없으면 빈칸) |

행 클릭 → 해당 담당자의 본인 주간 KPI 화면으로 이동 (관리자만).

#### 3.2.4 난이도별 SR 분포

데이터: `SUM(sr_completed GROUP BY difficulty_score)` 필터 적용

표시: 5→1 점 순서, 가로 막대 + 건수 + 비율 (%).

#### 3.2.5 이슈·알림 패널

자동 감지 항목 (있을 때만 표시):

- **미매핑 SR**: `sr_difficulty_mappings` 누락된 완료 SR이 있을 때
- **재오픈 SR**: `SUM(sr_reopened) > 0`
- **SR 미연결 커밋 50% 초과 담당자**: `sr_linked_commits / git_commits < 0.5`인 담당자 명단
- **동일 파일 반복 임계 초과 담당자**: §5.4 임계 1 초과 명단
- **시행착오 키워드 매칭 다수 담당자**: 매칭 ≥ 5인 명단

#### 3.2.6 코칭 후보 패널 (§5.5 매니저용)

**원칙**: 본 패널은 **지원·코칭 신호**이며 비난·평가 도구가 아닙니다. 코칭 워크플로우 진입 조건(2주 연속 🟡 또는 단일 주 🟠)을 충족하고, **본인이 매니저 알림에 동의한 경우에만** 표시됩니다.

```
┌─────────────────────────────────────────────────────────────────┐
│ 💙 코칭 후보 (본인 동의 알림)                                    │
│                                                                 │
│ • 김지원 — 2주 연속 🟡 주의                                       │
│   감지: Commit Efficiency 8.5, 재오픈 3건                       │
│   요청: [도메인 멘토 매칭]                                       │
│   [1:1 코칭 일정 잡기]                                           │
│                                                                 │
│ • 이준호 — 이번 주 🟠 코칭 권장                                   │
│   감지: 4가지 신호 모두 충족                                     │
│   요청: [페어 작업]                                              │
│   [1:1 코칭 일정 잡기]                                           │
└─────────────────────────────────────────────────────────────────┘
```

| 요소 | 동작 |
|---|---|
| 코칭 후보 명단 | 본인 동의 시에만 표시. 미동의자는 절대 노출 안 함 |
| 본인이 요청한 지원 형태 | 매니저 알림 / 도메인 멘토 / 페어 작업 중 본인 선택 |
| 1:1 코칭 일정 잡기 | 일정 도구 연결 (캘린더 모달 또는 외부 링크) |
| 종결 | 다음 주 신호 감소 또는 매니저 수동 종결로 명단에서 제거 |

**절대 금지** (§5.5.5):
- 코칭 후보를 일반 KPI 랭킹 표·이슈 알림 패널 등에 같이 표시 금지
- 본인 미동의 시 매니저 화면에 노출 금지 (감사 로그·정합성 점검 필수)
- "능률 저조 명단"·"부진 담당자" 등 평가 함의 용어 사용 금지 — 본 시스템에서는 일관되게 "코칭 후보"로 표기

#### 3.2.6 다운로드 (Word·Excel)

- Word: 팀 요약 + 담당자별 랭킹 + 알림 (`/team-weekly/.../download`)
- Excel: `weekly_kpi_snapshots` 전체 컬럼 CSV (분석 도구 입력용)

## 5. 데이터 흐름

```
[월요일 09:00 KST]
       ↓
주간 집계 잡 (Laravel Job·Schedule)
       ↓
1. SR 데이터 수집 (supportworks SR + sr_difficulty_mappings)
2. Git 데이터 수집 (git_commit_logs)
3. KPI 계산 (§5 산식)
4. 반복 커밋 페널티 계산 (§5.4)
       ↓
weekly_kpi_snapshots 테이블에 저장
       ↓
사용자 접속 시
       ↓
/my-weekly 컨트롤러
       ↓
weekly_kpi_snapshots 조회 (캐싱)
       ↓
화면 렌더
```

**캐싱**: 같은 주차 데이터는 변경 빈도 낮음 → Redis 또는 application 캐시(`Cache::remember`) 24시간.

**실시간성**: 본인이 SR 매핑·완료 시 즉시 반영하려면 모델 이벤트로 캐시 무효화.

## 7. 상호작용 (Interaction)

### 7.1 주차 네비게이션

- 이전/다음 화살표는 직전 12주까지 자유 이동
- 그 이전은 캘린더 모달 (월 단위 선택)
- 현재 주차(진행 중) 표시는 점선 테두리 (집계 미완 알림)

### 7.2 드릴다운

| 진입점 | 도착 |
|---|---|
| 완료 SR 행 | SR 상세 팝업 (기존 패턴) |
| 페널티 패널 항목 | 해당 파일/SR/커밋 상세 |
| 팀 랭킹 행 | 해당 담당자 본인 KPI 화면 |

### 7.3 빈 상태 (Empty State)

- 본 주차 미집계: "이번 주는 아직 집계되지 않았습니다 (월요일 09:00 KST 집계)"
- 데이터 없음: "이번 주에 처리한 SR이 없습니다"
- 미매핑 SR: 상단 노랑 알림 배너 + 매핑하기 버튼

### 7.4 반응형

- 데스크탑 (≥1280px): 8열 SR 지표 그리드, 7열 Git 지표
- 태블릿 (768~1279px): 4열×2행
- 모바일 (<768px): 2열×4행, 표는 가로 스크롤
