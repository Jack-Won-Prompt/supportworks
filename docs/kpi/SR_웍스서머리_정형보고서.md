# 웍스 서머리 정형 보고서 명세

> supportworks `/my-weekly` 의 **웍스 서머리 탭**에서 표시되는 SR 담당자 성과 보고서의 정형 양식 명세입니다. 기존의 AI 자유 생성 텍스트 방식은 폐기하고, [SR_담당자_주간성과지표.md](SR_담당자_주간성과지표.md)에 정의된 KPI 산식을 그대로 적용한 **정형(structured) 보고서**로 전환합니다.

## 참조 문서

- [SR_담당자_주간성과지표.md](SR_담당자_주간성과지표.md) — KPI 데이터·산식 (모든 산식의 원천)
- [SR_성과지표_화면구성.md](SR_성과지표_화면구성.md) — 본인/팀 화면 UI 명세
- [supportworks-env.md](supportworks-env.md) — 환경·디자인 토큰

## 1. 문서 개요

### 1.1 목적

기존 웍스 서머리는 AI(`WeeklyAiSummary`)가 자유 텍스트로 요약을 생성했습니다. 그러나 AI 자유 텍스트는:
- 산식 일관성·재현성 부족
- 페널티·코칭 신호 표준 표시 불가
- 정량 KPI 추적 어려움
- 운영팀 정량 평가 입력으로 사용 어려움

본 명세는 **정형(structured) 보고서**로 전환합니다 — 동일 데이터에 대해 항상 같은 양식으로 출력되어 추적·비교·평가가 가능하도록.

### 1.2 적용 범위

| 항목 | 변경 |
|---|---|
| 기존 type=full (지난달) | ✅ **정형 보고서로 교체** |
| 기존 type=this_month (이번달) | ✅ **정형 보고서로 교체** |
| 기존 type=weekly (주차) | ✅ **정형 보고서로 교체** |
| AI 자유 생성 텍스트 | ❌ **폐기** |
| `WeeklyAiSummary` 모델 | 호환성 유지 (양식만 정형으로). 또는 신규 `KpiReportSnapshot` 모델 |

## 2. 대상 화면

### 2.1 위치

기존 `/my-weekly` 의 두 번째 탭 **"웍스 서머리"** 그대로 사용.

- 라우트: 기존 [MyWeeklyReportController.php](../app/Http/Controllers/MyWeeklyReportController.php) 의 `index()` 유지
- 뷰: 기존 [my-weekly/index.blade.php](../resources/views/my-weekly/index.blade.php) `#tab-ai` 영역 확장
- AJAX 엔드포인트: 기존 `projects.weekly-reports.ai-summary.generate` / `.ai-summary` 라우트 패턴 그대로. 응답 본문이 AI 텍스트 → 정형 JSON으로 변경

### 2.2 권한

| 권한 | 조건 (기존 컨트롤러 패턴 재사용) |
|---|---|
| 탭 노출 | `$user->isAdmin() OR $user->is_sr_agent` (`canViewAiSummary`) |
| Git 동기화 버튼 | `$user->isAdmin() OR manager 역할` (`canSyncGit`) |
| 다운로드 (Word) | 보고서 생성 권한자와 동일 |

## 3. 컨트롤 패널 (기존 유지)

기존 웍스 서머리 컨트롤 패널의 구조는 유지하되, "서머리 생성" 버튼이 정형 보고서를 반환합니다.

```
┌─────────────────────────────────────────────────────────────────┐
│ [프로젝트] WITHWORKS ▼ 🔗                                        │
│ [기간 타입] 【지난달】 이번달 주차                                 │
│ [주차 선택] (weekly 선택 시) 월요일~금요일                        │
│ [SR 회사] ☑A사 ☑B사 ☐C사 (다중)                                  │
│ [담당자 필터] (옵션) 전체 / 본인 / 다중선택                       │
│ ────────────────────────────────────────────────────────────── │
│ [Git 동기화] (관리자·매니저)   [서머리 조회] [서머리 생성]        │
└─────────────────────────────────────────────────────────────────┘
```

### 3.1 컨트롤 동작

| 요소 | 동작 |
|---|---|
| 기간 타입 | full(지난달) / this_month(이번달) / weekly(특정 주차) 중 택1 |
| 주차 선택 | weekly 선택 시에만 활성화. 월요일~금요일 자동 |
| 프로젝트 | 기존 ProjectGitLink 기반. WITHWORKS 연결 표시 |
| SR 회사 | 기존 srCompaniesForFilter 재사용. 다중 선택 |
| **담당자 필터** (신규) | 기본은 SR 회사·프로젝트 멤버 전체. 운영자가 일부 선택 가능 |
| Git 동기화 | 기존 WithWorksGitController::sync. 보고서 생성 전 최신화 권장 |
| 서머리 조회 | 저장된 보고서 (KpiReportSnapshot) 재표시. AI 호출 없음 |
| 서머리 생성 | 정형 보고서 신규 산출 (시간 = SR + Git 집계) |

### 4.2 산출 한계 패널

KPI 산출에 영향을 미치는 입력 데이터 부재 사항을 **보고서 최상단에** 명시합니다. 비난 신호가 아닌 정합성 신호.

```
⚠️ 산출 한계
┌──────────────────────────────┬────────┬──────────────────────────┐
│ 항목                         │ 상태   │ 사유                      │
├──────────────────────────────┼────────┼──────────────────────────┤
│ SR 처리 실적 8 지표          │ ❌ 불가│ sr_difficulty_mappings   │
│                              │        │ 미구축 + [SR-xxxx] 0%    │
│ 종합 점수 (WeeklyScore)      │ ❌ 불가│ α(SR) 입력 부재          │
│ 품질 보정                    │ ❌ 불가│ SR 데이터 부재           │
│ 반복 커밋 페널티 캡(20%)     │ ⚠ 미적용│ WeeklyScore 부재         │
│ Git 7 지표                   │ ✅ 산출│ git log 실측             │
│ §5.4.1 반복 커밋 페널티      │ ✅ 산출│ (캡 미적용 raw 페널티)   │
└──────────────────────────────┴────────┴──────────────────────────┘
```

데이터 바인딩:
- 표는 보고서 생성 시점에 동적 결정 (예: SR 매핑 테이블이 생기면 상태 ✅로 자동 전환)
- 각 항목은 일관된 식별자(`sr_metrics`, `weekly_score`, `quality_correction`, `penalty_cap`, `git_metrics`, `repeat_penalty`)로 관리

### 4.3 담당자별 보고서 (1명당 반복)

각 담당자마다 다음 구조 반복:

```
─────────────────────────────────────────────────────────────────
N. {담당자명} ({git_author_id})
─────────────────────────────────────────────────────────────────

[Git 처리 실적]
┌──────────────────┬──────────┐
│ 커밋 수 (no-merges)│   105   │
│ 추가 LOC         │ +10,479  │
│ 삭제 LOC         │  -4,510  │
│ 순 LOC           │ +5,969   │
│ 변경 파일 (누적) │    863   │
│ SR 연결 커밋     │  0 (0%)  │
│ Commit Efficiency│ 산출 불가│
└──────────────────┴──────────┘

[반복 커밋 페널티 (§5.4.1)]
┌──────────────┬─────────────────────────────────┬──────┬────────┐
│ 패턴         │ 항목                            │ 횟수 │ 페널티  │
├──────────────┼─────────────────────────────────┼──────┼────────┤
│ #1 동일 파일 │ app/Helpers/DataHeader.php      │  28  │ -4.90🔴│

│ ...                                                              │
│ #3 시행착오  │ —                               │   0  │   0    │
│ 합계 (캡 적용 전)                                       │ -8.78  │
└─────────────────────────────────────────────────────────────────┘

[AI 활용 능률 신호 (§5.5)] — 본인 또는 관리자에게만
등급: 🟡 주의 (4가지 신호 중 2가지 발생)
   ⚠ Commit Efficiency 산출 불가 (SR 매핑 부재로 단독 판단 보류)
   ⚠ 반복 페널티 누적 -8.78 ≤ -0.3 임계 초과
   💡 힌트: ...

[코멘트 (코칭 관점)]
• DataHeader.php 28회 수정은 누진 페널티가 가장 강하게 발생...
• lang JSON 다국어 파일 다수 동시 5회 수정 — 자동화 검토 권장
• 시행착오 키워드 0건 — 깔끔한 메시지 관리는 강점
```

#### 4.3.1 Git 처리 실적 표 — 데이터 바인딩

| 라벨 | 컬럼·산식 |
|---|---|
| 커밋 수 (no-merges) | `git_commits` (`git log --no-merges --author=A --since=S --until=U \| wc -l`) |
| 추가 LOC | `git_added_loc` (shortstat 합산) |
| 삭제 LOC | `git_deleted_loc` |
| 순 LOC | `added - deleted` |
| 변경 파일 (누적) | `git_files` (shortstat 합산) |
| SR 연결 커밋 | `sr_linked_commits + "(" + percent + "%)"` (`[SR-xxxx]` 매칭) |
| Commit Efficiency | SR 매핑 있을 때만 — `median(commits_per_sr) + " commit/SR"`. 없으면 `"산출 불가 (SR 매핑 부재)"` |

#### 4.3.2 반복 커밋 페널티 표 — 데이터 바인딩

| 컬럼 | 데이터 |
|---|---|
| 패턴 | `#1 동일 파일` / `#2 동일 SR 커밋 폭증` / `#3 시행착오 키워드` |
| 항목 | 파일 경로 / SR ID / 키워드 매칭 커밋 SHA |
| 횟수 | 패턴별 카운트 |
| 페널티 | §5.4.1 산식 결과 + 등급 이모지(🔵🟡🟠🔴) |
| 합계 행 | 총합 (캡 적용 전 raw 페널티) |

표 정렬: **#1 → #2 → #3 순서**, 각 패턴 내에서는 페널티 큰 순 정렬.

#### 4.3.3 AI 활용 능률 신호 (§5.5)

기존 [SR_성과지표_화면구성.md §2.2.8](SR_성과지표_화면구성.md)의 패널을 보고서에 정형화. 단:

- **표시 대상**: 보고서 보는 사람이 **본인 또는 관리자**인 경우만
- **개인 비교 금지** (§5.5.5): 다른 담당자와 동일 보고서에서 비교 형태로 표시 금지. 본인 보고서 안에서만 표시
- SR 매핑 부재 시 등급 표시하되 "단독 판단 보류" 명시

#### 4.3.4 코멘트 (코칭 관점)

자동 생성되는 **자연어 코멘트** 3~5개 bullet. 다음 규칙으로 생성:

| 트리거 | 코멘트 템플릿 |
|---|---|
| 단일 파일 ≥10회 + 자체 시스템 핵심 파일 (`DataHeader.php`, `IntegrationController.php`, helpers) | "{파일} {N}회 수정은 자체 시스템 핵심 헬퍼(난이도 표 §13 점수 3 영역). 작업 단위 분할·사전 영향 분석 권장 신호" |
| 단일 파일 ≥10회 + UDI·외부 시스템 파일 | "{파일}는 fulfillment 난이도 표 §13 5점 영역(외부 API + 규제). Claude 단독 작성이 어려운 영역으로 시행착오가 정상 범위 가능" |
| lang JSON 다수 동시 변경 | "lang JSON 다국어 파일 다수 동시 N회 수정 — 다국어 일괄 동기화 자동화 검토 권장" |
| 시행착오 키워드 = 0 | "시행착오 키워드 0건 — 깔끔한 메시지 관리는 강점" |
| "원복" 키워드 ≥3 | "'원복' 시리즈를 구체적 사유 메시지로 대체 권장 (페널티 직접 영향 + 추적성 ↑)" |
| 시행착오 키워드 1~2 + 외부 API 영역 | "시행착오 키워드 N건은 양호한 편" |
| SR 연결 커밋 = 0% | "[SR-xxxx] prefix 미사용 — SR ↔ Git 추적성 확보 위해 컨벤션 적용 권장" |

코멘트는 **사실 기반·코칭 톤** 유지. 평가·비난 어휘 금지 (§5.5.5).

```

데이터 바인딩:
- 행: 보고서 §4.3 각 담당자별 산출값
- 정렬: **고정 컬럼 순서** (담당자 이름 가나다 또는 ID 순) — 매번 동일 순서로 비교 가능
- 강조: 가장 큰 값(최대) 또는 가장 작은 값(최소)에 굵게/색상 부각

**중요**: 본 표는 §5.5.5 "개인 명단 공개 비교" 제한과 별개로, **사실 수치 비교**이므로 허용. 단 능률 신호 등급은 본 표에 포함하지 않음 (능률 신호는 §4.3 개인 보고서에만).

• SR ↔ Git 연결 컨벤션 미적용 — N명 모두 [SR-xxxx] prefix 0% 
• WeeklyScore 산출 불가 — α(SR) 부재로 종합 점수 무의미
• 페널티 캡(20%) 미적용 — raw 페널티 절대값 표시
• 도메인 특성 미반영 — UDI 같은 5점 영역은 정상 범위 가능

5.2 명세서 적용 전 선결 과제
• supportworks sr_difficulty_mappings·weekly_kpi_snapshots 테이블 마이그레이션
• 커밋 메시지 [SR-xxxx] prefix 운영팀 합의·공지
• 페널티 산정 시 fulfillment 난이도 점수와 결합

5.3 페널티 산정의 정성적 활용 (현 시점 가능한 코칭 신호)
• 허정빈: DataHeader.php 같은 자체 시스템 핵심 파일은 사전 영향 분석 권장
• 진세종: "원복" 시리즈를 구체적 사유 메시지로 대체 권장
• 정현우: UDI 같은 외부 의존 영역은 페널티가 정상 범위일 수 있음

5.4 본 보고서의 위치
본 페널티 수치는 명세서 §5.4.4·§5.5.5에 따라 "코칭 신호"이지 평가·비난이 아닙니다.
SR 데이터 부재로 종합 점수 산출이 불가능한 현 단계에서는 
정성적 개선 신호로만 활용하시기 바랍니다.
```

데이터 바인딩:
- **5.1 측정의 한계**: §4.2 산출 한계 패널의 ❌·⚠ 항목을 자연어로 풀어 쓴 형태. 항목 변경 시 자동 갱신
- **5.2 선결 과제**: 산출 한계와 1:1 대응 — 한계가 해소되면 해당 항목 자동 제거
- **5.3 정성적 활용**: 각 담당자 §4.3 코멘트의 **첫 번째 bullet**을 추출해서 요약
- **5.4 본 보고서의 위치**: 고정 문구 (편집 불가)

## 5. 데이터 수집·집계 절차

### 5.1 기간 타입별 경계

| type | 시작 | 종료 |
|---|---|---|
| `full` (지난달) | 직전 달 1일 00:00 KST | 직전 달 말일 23:59 KST |
| `this_month` (이번달) | 이번 달 1일 00:00 KST | 산출 시점 (실시간) |
| `weekly` (주차) | 선택 주차 월요일 00:00 KST | 선택 주차 금요일 23:59 KST (또는 일요일 — 운영 결정) |

### 5.2 데이터 흐름

```
1. 사용자가 [서머리 생성] 버튼 클릭
   ↓
2. 백엔드: 기간 경계 + 대상 담당자 결정
   ↓
3. SR 데이터 조회 (supportworks DB)
   - sr_difficulty_mappings JOIN
   - 미매핑 SR은 산출 한계 패널에 기록
   ↓
4. Git 데이터 조회 (git_commit_logs 또는 실시간 git log)
   - 담당자별 커밋·LOC·파일 집계
   - 동일 파일 반복 카운트
   - 시행착오 키워드 매칭
   - [SR-xxxx] prefix 매칭
   ↓
5. 페널티 산출 (§5.4.1)
   - 누진 페널티 계산
   - 캡 적용 (WeeklyScore 있을 때만)
   ↓
6. 능률 신호 등급 (§5.5)
   - 본인·관리자에게만 표시
   ↓
7. 코멘트 자동 생성 (§4.3.4 룰)
   ↓
8. 정형 보고서 JSON 응답
   ↓
9. (선택) KpiReportSnapshot 저장 → 재조회 시 캐시
```

## 9. 한계·주의사항

- **AI 자유 생성 폐기로 인한 유연성 감소**: 정형 보고서는 일관성·재현성을 얻지만, 특수 케이스의 자유 서술이 어려움. 필요 시 §4.3.4 코멘트 자동 생성 룰 확장으로 보완
- **데이터 부재 시 보고서 의미 약화**: SR 매핑이 적용되기 전에는 Git 실적만 표시되어 본 명세서가 정의한 "SR + Git 두 축" 균형이 미달성. §4.2 산출 한계 패널이 이를 명시함
- **코멘트 자동 생성 룰의 한계**: §4.3.4 룰은 패턴 기반이라 도메인 깊은 분석은 어려움. 운영 데이터로 룰 보정 필요
- **권한·심리적 안전**: 능률 신호(§5.5)는 본인 동의 없이 매니저에 노출 금지. 감사 로그 권장
- **본 명세서는 화면 양식 정의이며, 평가 도구가 아닙니다.** §5.5.5 모든 금지 사항이 본 보고서에도 동일 적용됩니다.
