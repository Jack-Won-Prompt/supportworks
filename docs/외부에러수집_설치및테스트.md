# 외부 시스템(withworks 등) 에러 수집 — 설치 및 테스트

withworks 의 PHP/JS 에러를 supportworks 에 API 로 전달하고, SR 담당자/관리자가 통합 관리하는 흐름의 운영 가이드.

## 1. 아키텍처 한눈에

```
┌─────────────────────────────────────────┐         ┌──────────────────────────────────────┐
│ withworks                               │         │ supportworks                         │
│                                         │         │                                      │
│  PHP 예외 ──→ app/Exceptions/Handler    │         │  POST /api/external-errors           │
│              └─ SupportworksErrorReporter│ ─HMAC→ │   ↓ VerifyHmacSignature             │
│                                         │         │   ↓ ExternalErrorController          │
│  JS error  ──→ window.onerror           │         │   ↓ SystemErrorLog::recordExternal   │
│              └─ /js-errors              │         │                                      │
│                └─ JsErrorReporterCtrl   │         │  GET /user/system-errors             │
│                  └─ SupportworksError…  │ ─HMAC→ │   ↓ RequireSrOrAdmin (SR or Admin)   │
│                                         │         │   ↓ User\SystemErrorController       │
│                                         │         │   ↓ resources/views/user/...         │
└─────────────────────────────────────────┘         └──────────────────────────────────────┘
```

## 2. supportworks 측 설치

### 2.1 마이그레이션 실행

```bash
cd /e/xampp/htdocs/supportworks
php artisan migrate
```

`system_error_logs` 테이블에 `source` 와 `origin` 컬럼 추가.

### 2.2 .env 설정

`.env` 에 추가 (32바이트 이상 무작위 hex 권장):

```env
# withworks 와 공유하는 HMAC secret
WITHWORKS_HMAC_SECRET=여기에_충분히_긴_랜덤_문자열_64자_이상
# fulfillment 도 보낼 거면 별도 키
# FULFILLMENT_HMAC_SECRET=...
```

생성 예시:
```bash
php -r "echo bin2hex(random_bytes(32));"
```

### 2.3 캐시 클리어

```bash
php artisan config:clear
php artisan route:clear
```

### 2.4 권한 부여

대상 사용자가 SR 담당자라면 DB 에서 `users.is_sr_agent = 1` 설정.

```sql
UPDATE users SET is_sr_agent = 1 WHERE email = 'sr_user@example.com';
```

관리자(`role = 'admin'`)는 자동으로 접근 가능.

## 3. withworks 측 설치

### 3.1 .env 설정

`.env` 에 추가 (supportworks 의 `WITHWORKS_HMAC_SECRET` 과 **동일한 값**):

```env
SUPPORTWORKS_REPORT_ENABLED=true
SUPPORTWORKS_API_URL=http://localhost/supportworks/api/external-errors
SUPPORTWORKS_HMAC_SECRET=여기에_supportworks와_동일한_값
SUPPORTWORKS_SOURCE_KEY=withworks
SUPPORTWORKS_TIMEOUT=3
SUPPORTWORKS_CONNECT_TIMEOUT=2
```

배포 환경에서는 `SUPPORTWORKS_API_URL` 을 실제 supportworks 호스트로.

### 3.2 캐시 클리어

```bash
cd /e/xampp/htdocs/withworks
php artisan config:clear
php artisan route:clear
php artisan view:clear
```

## 4. 동작 테스트

### 4.1 HMAC 엔드포인트 수동 호출 (curl)

`WITHWORKS_HMAC_SECRET` 을 `mysecret` 으로 가정한 예시.

```bash
# Linux/macOS/Git Bash
SECRET="mysecret"
TS=$(date +%s)
BODY='{"level":"error","origin":"server","exception":"TestException","message":"Curl test","file":"/tmp/x.php","line":42,"context":{"url":"https://test.local/"}}'
SIG=$(printf "%s\n%s" "$TS" "$BODY" | openssl dgst -sha256 -hmac "$SECRET" -hex | awk '{print $2}')

curl -i \
  -H "Content-Type: application/json" \
  -H "X-WW-Source: withworks" \
  -H "X-WW-Timestamp: $TS" \
  -H "X-WW-Signature: sha256=$SIG" \
  -d "$BODY" \
  http://localhost/supportworks/api/external-errors
```

성공 시 `201 {"id":...}`.
잘못된 서명·만료된 timestamp 는 `401 Unauthorized`.

### 4.2 withworks 서버 예외 발생시켜 보기

withworks 의 아무 컨트롤러 라우트에 임시로:

```php
throw new \RuntimeException('서버 예외 테스트 ' . now());
```

브라우저로 해당 경로 진입 → withworks 의 Handler.report() 가 SupportworksErrorReporter 호출 → supportworks 에 기록.

### 4.3 withworks JS 에러 발생시켜 보기

withworks 페이지의 브라우저 콘솔에서:

```javascript
setTimeout(() => { throw new Error('JS 클라이언트 에러 테스트'); }, 0);
```

→ `/js-errors` POST → JsErrorReporterController → supportworks 로 forward.

### 4.4 supportworks 에서 확인

1. SR 담당자 또는 admin 계정으로 supportworks 로그인
2. 좌측 사이드바 "SR 담당자" 섹션 → "시스템 에러 관리" 클릭
3. 출처 칩 (withworks / supportworks / fulfillment) 으로 필터링
4. 행 클릭 → 상세 페이지로 이동, "해결됨으로 표시" 가능
5. Admin 권한자만 삭제 버튼 노출

## 5. 권한 매트릭스

| 작업                   | 비로그인 | Member | SR(`is_sr_agent`) | Admin |
| --------------------- | :------: | :----: | :--------------: | :---: |
| 사이드바 메뉴 노출       | ✗        | ✗      | ✓                | ✓     |
| 목록 조회               | ✗        | ✗      | ✓                | ✓     |
| 상세 조회               | ✗        | ✗      | ✓                | ✓     |
| 해결됨 표시             | ✗        | ✗      | ✓                | ✓     |
| 일괄 해결됨 표시         | ✗        | ✗      | ✓                | ✓     |
| 개별 삭제               | ✗        | ✗      | ✗                | ✓     |
| 해결된 항목 일괄 삭제     | ✗        | ✗      | ✗                | ✓     |
| 알림 수신 (FCM/Mail/SMS) | ✗      | ✗      | ✓                | ✗\*   |

\* Admin 은 **내부** supportworks 에러는 기존 `notifyAdmins` 로 알림 받지만, 외부(withworks 등) 에러 알림은 SR 전용으로 분리.

기존 admin 영역(`/admin/system-errors`) 메뉴는 그대로 유지 — 관리자는 양쪽 메뉴 모두 접근 가능.

## 5.1. 외부 에러 알림 (SR 담당자 자동 통지)

외부 시스템에서 들어온 에러는 SR 담당자에게 자동 통지됨.

- **수신자**: `users.is_sr_agent = 1` 인 모든 사용자
- **레벨**: `error / critical / alert / emergency` (warning/info 제외)
- **채널**: FCM (디바이스 토큰 있을 때) + Email + SMS (전화번호 있을 때)
- **쿨다운**: 같은 source+exception+file+line 조합은 5분간 재발송 안 함 (도배 방지)
- **장애 격리**: 채널별 독립 try-catch — 한 채널 실패가 다른 채널 막지 않고, 알림 실패가 에러 로그 자체 저장을 막지 않음

### 발송 채널별 사전 조건

| 채널  | 사전 조건                                        | 확인 방법 |
| ----- | ----------------------------------------------- | -------- |
| FCM   | 사용자 디바이스 토큰이 등록되어 있어야 함         | `device_tokens` 테이블 |
| Email | `mail.default` 가 실제 발송 드라이버 (smtp/ses 등) | `php artisan tinker` → `config('mail.default')` |
| SMS   | POPBILL 계정 + SR `users.phone` 입력            | `config('popbill.LinkID')` 가 SET 인지 확인 |

⚠️ **개발 환경**: `MAIL_MAILER=log` 인 경우 메일은 실제로 발송되지 않고 `storage/logs/laravel.log` 에만 기록됨.
운영 배포 시 SMTP/SES 로 전환 필요.

### SMS 본문 포맷
`[SW에러:LEVEL/source] message...` (최대 ~90byte, 초과 시 자동 LMS 전환)

## 6. 페이로드 스펙

POST `/api/external-errors` 요청 본문:

```jsonc
{
  "level":     "error",         // emergency|alert|critical|error|warning|notice|info|debug (옵션, 기본 error)
  "origin":    "server",        // server|client|console|job (옵션, 기본 server)
  "exception": "RuntimeException", // 옵션
  "message":   "에러 메시지",     // 필수, 최대 65000자
  "file":      "/app/.../X.php",
  "line":      42,
  "trace":     "stack trace…",
  "context":   {                // 자유 JSON
    "url":        "...",
    "method":     "GET",
    "ip":         "...",
    "user_id":    123,
    "user_name":  "홍길동",
    "user_agent": "..."
  }
}
```

응답: `201 {"id": 12345}` / `422 invalid message` / `401 unauthorized` (HMAC 실패)

**`source` 는 body 가 아니라 HMAC 헤더(`X-WW-Source`)로만 결정.** body 의 source 는 무시되어 스푸핑 방지.

## 7. 운영 노트

- **fire-and-forget**: withworks 가 supportworks 에 닿지 못해도 본 시스템(withworks)은 정상 동작. `Log::warning` 만 남음.
- **타임아웃**: 기본 connect 2s / total 3s. 응답 지연이 withworks 응답 속도에 영향 가지 않도록 설계.
- **노이즈 필터**: 401/403/404/419/422, ValidationException, TokenMismatch, AuthenticationException, ModelNotFound 는 supportworks 에 보내지 않음.
- **JS 세션당 30건 제한**: 같은 페이지에서 무한 루프 에러가 도배되는 걸 방지.
- **키 회전**: secret 교체 시 양쪽 .env 를 거의 동시에 갱신하면 약간의 401 발생 가능. 무중단 회전을 원하면 supportworks 측 `config/services.php` 의 `sources` 에 새 소스 키를 임시 추가하고 점진 전환.

### 7.1. dev 환경(xampp 단일 Apache) 동거 한계 — 2026-05-28 검증

같은 xampp Apache + mod_php 에서 withworks 와 supportworks 가 함께 호스팅되면 **web→web HTTP 호출 시 환경변수 오염**으로 supportworks 가 withworks 의 DB(`warehouse`)를 보고 INSERT 시도 → 500.

- 추정 원인: phpdotenv `createImmutable` + Apache worker 의 process-wide `getenv()` 잔존. withworks 가 먼저 set 한 env 를 supportworks 부팅이 못 덮어씀.
- **CLI/별도 process는 정상**: `php artisan tinker` 직접 호출, curl 외부 호출은 정상 동작.
- 따라서 dev 에서는 `.env` 의 `SUPPORTWORKS_REPORT_ENABLED=false` 권장.

### 7.2. 운영 분리 vhost 권장 설정

운영(또는 동거 회피)에서는 두 시스템을 별도 vhost 로 분리. xampp `httpd-vhosts.conf` 예:

```apache
<VirtualHost *:80>
    ServerName withworks.local
    DocumentRoot "E:/xampp/htdocs/withworks/public"
    <Directory "E:/xampp/htdocs/withworks/public">
        AllowOverride All
        Require all granted
    </Directory>
</VirtualHost>

<VirtualHost *:80>
    ServerName support.local
    DocumentRoot "E:/xampp/htdocs/supportworks/public"
    <Directory "E:/xampp/htdocs/supportworks/public">
        AllowOverride All
        Require all granted
    </Directory>
</VirtualHost>
```

`hosts` 파일에:
```
127.0.0.1   withworks.local
127.0.0.1   support.local
```

이후 withworks 의 `.env`:
```
SUPPORTWORKS_REPORT_ENABLED=true
SUPPORTWORKS_API_URL=http://support.local/api/external-errors
```

vhost 분리로 worker 격리 → 환경변수 오염 해결.

### 7.3. 비동기 대안 (필요 시)

vhost 분리가 어려운 dev 환경에서 즉시 검증이 필요하면 `SupportworksErrorReporter::send()` 를 Queue dispatch 로 비동기화. 별도 worker process 에서 실행되므로 환경 격리. 단, withworks v1 에 queue 인프라 구성이 별도 작업.

## 8. 확장 — fulfillment 도 보내려면

1. fulfillment 의 `Handler.php` 에 동일한 후크 추가 (`SupportworksErrorReporter::fromThrowable`).
2. `fulfillment/config/services.php` 의 `supportworks.source` 를 `fulfillment` 로.
3. `fulfillment/.env` 의 `SUPPORTWORKS_HMAC_SECRET` 을 supportworks 의 `FULFILLMENT_HMAC_SECRET` 과 동일한 값으로.
4. supportworks `.env` 의 `FULFILLMENT_HMAC_SECRET` 설정.

`source` 칩으로 시스템별 필터링.
